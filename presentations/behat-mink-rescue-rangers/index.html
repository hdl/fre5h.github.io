A PHP framework for testing your business expectations

Konstantin Kudryashov

Creator of Behat and Mink
You might think about Behat as a tool to teach your clients about testing, but in reality it's a tool to teach you about client business.
  • Behat був розроблений під вараженням від Cucumber
  • PHP-фреймворк (>=PHP 5.3) для тестування поведінки додатку
  • Gherkin DSL - простий синтаксис для написання історій
  • Зручне написання тестів за допомогою аннотацій
  • Тестування будь-чого, в тому числі і взаємодія з браузерами за допомогою Mink

Установка

  • Composer
  • PHAR
  • Git

Composer

  1. Створюємо файл composer.json в директорії проекту
    {
    "require": {
        "behat/behat": "2.4.*@stable"
        },
    "minimum-stability": "dev",
    "config": {
        "bin-dir": "bin/"
    }
}
  2. Завантажуємо composer.phar і запускаємо з командою install 
    $ curl http://getcomposer.org/installer | php
$ php composer.phar install
  3. І можемо користуватись
    $ bin/behat

PHAR

$ wget https://github.com/downloads/Behat/Behat/behat.phar
$ php behat.phar

Git

  1. Клонуємо з GitHub
    $ git clone git://github.com/Behat/Behat.git && cd Behat
$ git submodule update –init
  2. Завантажуємо composer.phar і запускаємо з командою install 
    $ wget -nc http://getcomposer.org/composer.phar
$ php composer.phar install
  3. І можемо користуватись
    $ bin/behat

Ініціалізація

  1. Виконуємо команду
    $ behat --init
  2. Структура каталогів
    |-- features
   `-- bootstrap
      `-- FeatureContext.php
    • features/ - Файли *.feature, які буде виконувати Behat
    • features/bootstrap/ - Всі файли *.php з цієї директорії будуть автоматично завантажені перед запуском тестів
    • features/bootstrap/FeatureContext.php - Файл контексту, тут будуть знаходиться методи для ваших кроків

Конфігурація

  • По замовчуванню конфігурація зберігається в файлі behat.yml
  • При потребі можна запустити з іншою конфігурацією:
    $ behat --config custom-config.yml
  • Приклад конфігурації: 
    default:
    paths:
        features: features
        bootstrap: %behat.paths.features%/bootstrap

check_errors:
    filters:
        tags: "@error"
    formatter:
        name: progress

ci:
    formatter:
        name:       junit
        parameters:
            output_path: /var/tmp/junit
  • Запускаємо з різними профілями:
    $ behat --profile check_errors
$ behat --profile ci

Gherkin DSL

Feature: Describe some feature of application
    In order to realize a named business value
    As a some user
    I need to be able to do some action

    Background:
        Given prepare system to known stage

    Scenario: Check some business logic of application
        Given some precondition
          And some other precondition
         When some action by user
          And some other action
         Then some testable outcome is achieved
          And something else we can check happens too
          But something should not happen

    In order to ... Цінність фунціоналу
    As a ... Певний користувач
    I need ... Скористатись певним функціоналом

    Background:

    Scenario: ... Перевірка певної бізнес-логіки додатку

Ось що видасть вам Behat для matrix.feature

/** @Given /^there is Morpheus$/ */
public function thereIsMorpheus() { throw new PendingException(); }
/** @Given /^there is Neo$/ */
public function thereIsNeo() { throw new PendingException(); }
/** @Given /^there is a blue pill$/ */
public function thereIsABluePill() { throw new PendingException(); }
/** @Given /^there is a red pill$/ */
public function thereIsARedPill() { throw new PendingException(); }
/** @When /^Neo eat red pill$/ */
public function neoEatRedPill() { throw new PendingException(); }
/** @Then /^he should stay in Wonderland and see how deep the rabbit-hole goes$/ */
public function heShouldStayInWonderlandAndSeeHowDeepTheRabbitHoleGoes() { throw new PendingException(); }
/** @Given /^the story should not end$/ */
public function theStoryShouldNotEnd() { throw new PendingException(); }

Кожен крок - це метод

/**
 * Feature context
 */
class FeatureContext extends MinkContext
{
    /**
     * Go to the homepage and check that it is enabled
     * and returns status code 200
     *
     * @Given /^I am on the homepage$/
     */
    public function iAmOnHomepage()
    {
        $this->visit('/');
        $this->assertResponseStatus(200);
    }
}

@Given, @When, @Then

  • Кроки описуються в файлі FeatureContext.php
  • @And, @But - допоміжні кроки для спрощення речень
  • Behat відноситься до всіх кроків однаково, але ви повинні по різному!
  • Якщо крок не згенерував exception - він пройшов успішно
  • Використовуємо assert-методи з PHPUnit

Запуск тестів з консолі

  • Запустити всі тести: 
    $ behat
  • Запустити окрему фічу: 
    $ behat features/some.feature
    або 
    $ behat --name="Feature name"
  • Запустити окремий сценарій з деякої фічі: 
    $ behat features/some.feature:15

          And I press "Login"

MetaSteps

Об’єднуємо методи в один

/**
 * Login as partner
 *
 * @Given /^I login as partner$/
 */
public function iLoginAsPartner()
{
    $this->iAmOnPartnerLoginPage();
    $this->fillField('_username', 'partner@test.com');
    $this->fillField('_password', '1234');
    $this->pressButton('Login');
    $this->iShouldBeAuthenticatedAsPartner();
}

Hooks

  • @BeforeSuite, @AfterSuite - виконуються для всіх фіч
  • @BeforeFeature, @AfterFeature - виконуються для кожної фічі
  • @BeforeScenario, @AfterScenario - виконуються для кожного сценарію
  • @BeforeStep, @AfterStep - виконується для кожного кроку
/** @BeforeFeature */
public static function setupFeature(FeatureEvent $event)
{
    // prepare something
}

/** @AfterFeature */
public static function teardownFeature(FeatureEvent $event)
{
    // clean something
}

Tags

  • @blog
    Feature: Blog
        ...

    @login
    Scenario: User login
        ...
  • Можна запускати певні сценарії: 
    $ behat --tags '@login,@logout'
$ behat --tags 'login,error'
$ behat --tags '@login&&@admin'
$ behat --tags '~@error'
  • Можна кастомізувати хуки для конкретних сценаріїв: 
    /**
 * Load fixtures for Voucher feature
 *
 * @BeforeScenario @voucher-feature
 */
public function loadFixturesForVoucherFeature()
{
    $this->loadFixtures('Path\To\Fixtures\VoucherFeature');
}

Acceptance test framework for web applications

  • PHP-фреймворк (>=PHP 5.3) для тестування веб-додатку
  • Надає API для роботи з різними емуляторами браузерів
  • Підтримувані емулятори браузерів:
    • Goutte
    • Zombie.js
    • Selenium
    • Selenium2
    • Sahi

Установка

  • Приклад composer.json для встановлення Mink
    {
    "require": {
        "behat/mink": "*",

        "behat/mink-goutte-driver": "*",
        "behat/mink-selenium-driver": "*",
        "behat/mink-selenium2-driver": "*",
        "behat/mink-sahi-driver": "*",
        "behat/mink-zombie-driver": "*"
    }
}
  • Також доступна установка через PHAR і Git

Можливості Mink

  • Контроль браузера через об’єкт Session:
    • Навігація по сторінкам
    • Cookie
    • Заголовки
    • HTTP Authentication
  • Доступ до елементів сторінки через селектори:
    • Named
    • CSS
    • XPath
    • ﬁnd* Methods
  • Над вибраними елементами ми можемо робити:
    • Зчитувати значення атрибутів
    • Емулювати події браузера
    • Заповняти форми, відмічати чекбокси, приєдувати файли і т.д.

    {
    "require": {
        "behat/behat": "2.4.*@stable",
        "behat/mink": "1.4.*@stable",

        "behat/mink-extension": "*",

        "behat/mink-goutte-driver": "*",
        "behat/mink-selenium2-driver": "*"
    },
    "minimum-stability": "dev",
    "config": {
        "bin-dir": "bin/"
    }
}
  • MinkContext - містить готові кроки для керування елементами браузера
  • Mink також можна використовувати окремо
  • Для тестування JavaScript - потрібно додати тег @javascript
